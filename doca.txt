UI Signal	Backend Requirement
Email + password signup	POST /auth/register/email with bcrypt hashing
Phone signup with country code	POST /auth/register/phone → trigger SMS OTP
4-digit OTP verification screen	POST /auth/otp/verify with time-limited token (5 min TTL)
OTP resend link	POST /auth/otp/resend with rate limiting (3/hr)
Login with email/password	POST /auth/login → JWT access + refresh tokens
Forgot password link	POST /auth/password/reset-request + email link
8-step profile setup	PATCH /users/me/profile with partial update support, step tracking
Profile System (from profile-setup, account)
UI Signal	Backend Requirement
Name, Age fields	users.display_name, users.birth_date (never store raw age — calculate)
Gender single select (5 options)	user_profiles.gender enum
Sexual orientation single select (7 options)	user_profiles.orientation enum
Interested In multi-select	user_preferences.interested_in[] array
Relationship type single select	user_preferences.relationship_type enum
Languages multi-select	user_languages junction table
Interests multi-select (12 options)	user_interests junction table
Photo upload (final step)	Media upload → S3 → store URL in user_media
Account stats: matches, chats, likes	Materialized counters on user_stats table
Verified badge	users.is_verified boolean
Public/Private account toggle	user_profiles.visibility enum
Deactivate / Permanently delete	users.account_status enum + soft delete + data purge job
Discover Tab (from index.tsx — card swiping)
UI Signal	Backend Requirement
Profile cards with distance	PostGIS geo-query sorted by proximity
Like (❤️) / Pass (✗)	POST /swipes → creates swipe record, triggers match check
Mutual like → match animation	Match detection on every like: if reverse swipe exists → create match
Card shows age, bio, interests, verified	Single profile detail endpoint with joins
Cards auto-advance	Pagination cursor — serve N profiles per call
Explore Tab (from explore.tsx — mood browsing)
UI Signal	Backend Requirement
Mood selection (5 moods)	users.current_mood + GET /discover/explore?mood=...
Profile cards per mood	Filter profiles by active mood, proximity, preferences
Inbox (from inbox.tsx, chat-detail.tsx)
UI Signal	Backend Requirement
Vibed Profiles row (matches online)	GET /matches?online=true — matches with last_seen recently
Chat list with unread count	GET /conversations with aggregated unread_count
All / Unread tab filters	?filter=unread query param
Search conversations	?search= full-text query
Chat detail with message history	GET /conversations/:id/messages paginated
Read receipts (✓✓ blue)	PATCH /messages/:id/read → WebSocket broadcast
Typing indicator	WebSocket event typing:start / typing:stop
Message input max 500 chars	Validation at API layer
Attachment button	POST /conversations/:id/attachments → S3 upload
Phone/Video call (UI only)	Design-ready: WebRTC signaling endpoint
Upload Tab (from upload.tsx — 5-step flow)
UI Signal	Backend Requirement
Photo/Video selection	Client picks from library → multipart upload to S3 presigned URL
Crop step (square, portrait, landscape)	Crop hints stored; actual processing by Sharp/Lambda
Caption field	posts.caption text field
Mood selection	posts.mood enum field
Privacy toggle (Public/Private)	posts.visibility enum — enforced at feed query level
Review & publish	POST /posts
Account Tab (from account.tsx)
UI Signal	Backend Requirement
Block user / Unblock	POST /blocks / DELETE /blocks/:userId
Report user	POST /reports → abuse report record
Premium subscription button	GET /subscriptions/plans + POST /subscriptions/checkout
Logout	Invalidate refresh token in DB/Redis
Account deletion	DELETE /users/me → cascade soft delete + 30-day purge job

🧰 3. Tech Stack Selection & Justification
Backend Framework — NestJS (Node.js + TypeScript)
Why NestJS	Reasoning
Modular architecture	Each domain (auth, users, matches, chat) is an isolated NestJS module — critical for team scale
TypeScript-first	Matches frontend language, shared type contracts possible
Dependency injection	Testable, SOLID services out of the box
Ecosystem maturity	First-class support for Socket.io, BullMQ, Passport, TypeORM/Prisma, Swagger
Guards & Interceptors	JWT guards, role guards, rate limit guards are native patterns
Widely adopted for apps like this	Production use at Roblox, Adidas, Autodesk
Why not Express? Too barebones — requires manual architecture decisions at every layer. At scale, inconsistency creeps in.

Why not Python/FastAPI? Node.js event loop is superior for WebSocket-heavy apps. Shared TypeScript types with the RN frontend is a significant DX win.

Database — PostgreSQL 16 + PostGIS
Why PostgreSQL	Reasoning
ACID transactions	Match creation, payment processing, account deletion need guaranteed consistency
PostGIS extension	Native geo-spatial queries for find profiles within X km — essential for dating apps
JSONB columns	Future-ready for flexible preference fields without schema migrations
Full-text search	Native tsvector for conversation search
Row-level security	Per-row privacy enforcement possible at DB level
Mature tooling	pgAdmin, pgBadger, pg_stat_statements for performance insight
Why not MongoDB? Relational integrity is non-negotiable: matches reference two users, messages reference conversations, blocks reference users. MongoDB's lack of joins makes complex queries expensive and inconsistent.

Why not MySQL? PostGIS support and JSONB are significantly more capable in PostgreSQL.

Cache / Real-Time — Redis 7 (Cluster Mode)
OTP storage with TTL (auto-expiry, no cron needed)
JWT refresh token store + blacklist for logout
Rate limiting buckets per IP + per user
Socket.io adapter for horizontal WebSocket scaling across multiple API pods
Online presence (SETEX user:online:{id} 30 "1" — refreshed on heartbeat)
BullMQ job queue backing store (match notifications, media processing)
Profile card cache (invalidated on profile update)
Real-Time — Socket.io
Battle-tested for chat applications (WhatsApp Web level usage)
Automatic fallback from WebSocket → long-polling (important for unstable mobile connections)
Room-based architecture maps perfectly to conversations
First-class NestJS Gateway support
ORM — Prisma
Type-safe schema → generated TypeScript client (zero SQL injection risk at ORM layer)
Migration system with prisma migrate (version-controlled schema changes)
Excellent PostgreSQL + PostGIS support
Schema-as-code approach matches professional team workflows
Auth — JWT (access + refresh token rotation)
Access token: 15 min TTL, stateless verification
Refresh token: 30 day TTL, stored in refresh_tokens table (allows remote logout from all devices)
Refresh token rotation: on each use, old token revoked, new one issued
Phone OTP: 6-digit (upgraded from UI's 4-digit for security), stored in Redis with 5-min TTL, max 3 attempts
Media — AWS S3 + CloudFront CDN
Profile photos, post images/videos uploaded via presigned S3 URLs (client uploads directly — never through API server)
Sharp running in AWS Lambda processes images: resize, compress, generate thumbnails
CloudFront CDN delivers media at edge — critical for global dating app performance
Videos: S3 → AWS Elemental MediaConvert for transcoding
Message Queue — BullMQ
Match found → queue push notification job
Post published → queue feed fanout job
Account deleted → queue data purge job (30-day delay)
Premium expired → queue downgrade job
Media uploaded → queue Sharp processing job
API Style — REST (not GraphQL)
Why REST:

Mobile apps with fixed screens map cleanly to REST resources
HTTP caching (ETags, Cache-Control) works out of the box — critical for profile card caching
Rate limiting is simpler (per route per user)
Swagger/OpenAPI generation is first-class in NestJS REST
No over-fetching risk at our scale — each screen has a purpose-built endpoint
Why not GraphQL: N+1 query risk without DataLoader; increased complexity for mobile team; harder to rate-limit per operation; overkill for a well-defined mobile app UI.

-- ============================================================
-- EXTENSIONS
-- ============================================================
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "postgis";
CREATE EXTENSION IF NOT EXISTS "pg_trgm"; -- for fuzzy text search

-- ============================================================
-- ENUMS
-- ============================================================
CREATE TYPE account_status     AS ENUM ('active', 'inactive', 'suspended', 'deleted', 'pending_verification');
CREATE TYPE subscription_tier  AS ENUM ('free', 'premium', 'premium_plus');
CREATE TYPE gender_type        AS ENUM ('man', 'woman', 'non_binary', 'prefer_not_to_say', 'other');
CREATE TYPE orientation_type   AS ENUM ('straight', 'gay', 'lesbian', 'bisexual', 'pansexual', 'asexual', 'queer');
CREATE TYPE relationship_goal  AS ENUM ('serious', 'casual', 'friendship', 'not_sure');
CREATE TYPE interested_in_type AS ENUM ('men', 'women', 'non_binary', 'everyone');
CREATE TYPE visibility_type    AS ENUM ('public', 'private', 'matches_only');
CREATE TYPE swipe_action       AS ENUM ('like', 'pass', 'super_like');
CREATE TYPE match_status       AS ENUM ('active', 'unmatched', 'blocked');
CREATE TYPE message_type       AS ENUM ('text', 'image', 'video', 'audio', 'gif', 'system');
CREATE TYPE message_status     AS ENUM ('sent', 'delivered', 'read');
CREATE TYPE post_visibility    AS ENUM ('public', 'private', 'matches_only');
CREATE TYPE mood_type          AS ENUM ('lonely', 'happy', 'stressed', 'heartbroken', 'flirty');
CREATE TYPE report_reason      AS ENUM ('spam', 'inappropriate', 'harassment', 'fake_profile', 'underage', 'other');
CREATE TYPE media_purpose      AS ENUM ('profile_photo', 'post_image', 'post_video', 'message_attachment');
CREATE TYPE auth_provider      AS ENUM ('email', 'phone', 'google', 'apple');
CREATE TYPE notification_type  AS ENUM ('match', 'message', 'like', 'super_like', 'profile_view', 'system');

-- ============================================================
-- CORE USERS TABLE
-- ============================================================
CREATE TABLE users (
    id                  UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    display_name        VARCHAR(50)       NOT NULL,
    birth_date          DATE              NOT NULL, -- never store age, calculate it
    email               CITEXT            UNIQUE,   -- case-insensitive email
    phone               VARCHAR(20)       UNIQUE,
    phone_country_code  VARCHAR(5),
    auth_provider       auth_provider     NOT NULL DEFAULT 'email',
    password_hash       TEXT,                       -- NULL for phone-only auth
    account_status      account_status    NOT NULL DEFAULT 'pending_verification',
    subscription_tier   subscription_tier NOT NULL DEFAULT 'free',
    is_email_verified   BOOLEAN           NOT NULL DEFAULT FALSE,
    is_phone_verified   BOOLEAN           NOT NULL DEFAULT FALSE,
    is_verified         BOOLEAN           NOT NULL DEFAULT FALSE, -- identity verified (manual or ID check)
    is_admin            BOOLEAN           NOT NULL DEFAULT FALSE,
    setup_completed     BOOLEAN           NOT NULL DEFAULT FALSE, -- all 8 profile steps done
    setup_step          SMALLINT          NOT NULL DEFAULT 0,     -- which step they're on
    last_active_at      TIMESTAMPTZ,
    created_at          TIMESTAMPTZ       NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ       NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ,                              -- soft delete timestamp
    purge_at            TIMESTAMPTZ,                              -- scheduled hard delete (30 days after deleted_at)
    CONSTRAINT chk_email_or_phone CHECK (email IS NOT NULL OR phone IS NOT NULL),
    CONSTRAINT chk_age CHECK (birth_date <= NOW() - INTERVAL '18 years') -- 18+ enforcement
);

-- ============================================================
-- USER PROFILES (extended, separate table for clean separation)
-- ============================================================
CREATE TABLE user_profiles (
    user_id         UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    bio             VARCHAR(500),
    gender          gender_type,
    orientation     orientation_type,
    relationship_goal relationship_goal,
    visibility      visibility_type   NOT NULL DEFAULT 'public',
    current_mood    mood_type,
    location        GEOGRAPHY(Point, 4326),       -- PostGIS point (lng, lat)
    location_city   VARCHAR(100),                 -- human-readable city name
    location_country VARCHAR(100),
    max_distance_km SMALLINT          NOT NULL DEFAULT 50,
    show_distance   BOOLEAN           NOT NULL DEFAULT TRUE,
    show_age        BOOLEAN           NOT NULL DEFAULT TRUE,
    avatar_media_id UUID,                         -- FK to user_media (set after media table created)
    updated_at      TIMESTAMPTZ       NOT NULL DEFAULT NOW()
);

-- ============================================================
-- USER PREFERENCES (who they want to see)
-- ============================================================
CREATE TABLE user_preferences (
    user_id           UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    interested_in     interested_in_type[] NOT NULL DEFAULT '{}',
    age_min           SMALLINT NOT NULL DEFAULT 18,
    age_max           SMALLINT NOT NULL DEFAULT 45,
    CONSTRAINT chk_age_range CHECK (age_min >= 18 AND age_max >= age_min)
);

-- ============================================================
-- USER LANGUAGES (junction)
-- ============================================================
CREATE TABLE user_languages (
    user_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    language VARCHAR(50) NOT NULL,
    PRIMARY KEY (user_id, language)
);

-- ============================================================
-- USER INTERESTS (junction)
-- ============================================================
CREATE TABLE user_interests (
    user_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    interest VARCHAR(50) NOT NULL,
    PRIMARY KEY (user_id, interest)
);

-- ============================================================
-- USER MEDIA (photos and videos — profile + posts)
-- ============================================================
CREATE TABLE user_media (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    purpose         media_purpose NOT NULL,
    s3_key          TEXT NOT NULL,                -- raw S3 object key
    cdn_url         TEXT NOT NULL,                -- CloudFront URL (served to client)
    thumbnail_url   TEXT,                         -- for videos
    mime_type       VARCHAR(50) NOT NULL,
    file_size_bytes BIGINT NOT NULL,
    width_px        INTEGER,
    height_px       INTEGER,
    duration_secs   INTEGER,                      -- for videos
    display_order   SMALLINT NOT NULL DEFAULT 0, -- order within profile gallery
    is_primary      BOOLEAN NOT NULL DEFAULT FALSE,
    processing_done BOOLEAN NOT NULL DEFAULT FALSE, -- Lambda processing complete
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Add FK after user_media is created
ALTER TABLE user_profiles ADD CONSTRAINT fk_avatar FOREIGN KEY (avatar_media_id) REFERENCES user_media(id) ON DELETE SET NULL;

-- ============================================================
-- USER STATS (materialized counters — avoid expensive COUNT queries)
-- ============================================================
CREATE TABLE user_stats (
    user_id         UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    total_matches   INTEGER NOT NULL DEFAULT 0,
    total_likes_received INTEGER NOT NULL DEFAULT 0,
    total_chats     INTEGER NOT NULL DEFAULT 0,
    profile_views   INTEGER NOT NULL DEFAULT 0,
    activity_score  NUMERIC(5,2) NOT NULL DEFAULT 0.00, -- for matching algorithm
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ============================================================
-- REFRESH TOKENS (for auth)
-- ============================================================
CREATE TABLE refresh_tokens (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT NOT NULL UNIQUE, -- bcrypt hash of token
    device_id   VARCHAR(100),         -- track per-device sessions
    device_name VARCHAR(100),
    ip_address  INET,
    user_agent  TEXT,
    expires_at  TIMESTAMPTZ NOT NULL,
    revoked_at  TIMESTAMPTZ,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ============================================================
-- SWIPES (the core swiping mechanic)
-- ============================================================
CREATE TABLE swipes (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    swiper_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    swiped_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    action      swipe_action NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_swipe UNIQUE (swiper_id, swiped_id)
);

-- ============================================================
-- MATCHES (created when mutual like detected)
-- ============================================================
CREATE TABLE matches (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_a_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, -- always lower UUID alphabetically
    user_b_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, -- always higher UUID alphabetically
    status      match_status NOT NULL DEFAULT 'active',
    matched_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    unmatched_at TIMESTAMPTZ,
    unmatched_by UUID REFERENCES users(id),
    CONSTRAINT uq_match UNIQUE (user_a_id, user_b_id),
    CONSTRAINT chk_match_order CHECK (user_a_id < user_b_id) -- prevents duplicate (A,B) and (B,A)
);

-- ============================================================
-- CONVERSATIONS (one per match, but also supports group future)
-- ============================================================
CREATE TABLE conversations (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    match_id        UUID UNIQUE REFERENCES matches(id) ON DELETE CASCADE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_message_at TIMESTAMPTZ
);

-- ============================================================
-- CONVERSATION PARTICIPANTS (normalized)
-- ============================================================
CREATE TABLE conversation_participants (
    conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    joined_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_read_at    TIMESTAMPTZ,                      -- for unread count calculation
    is_archived     BOOLEAN NOT NULL DEFAULT FALSE,
    PRIMARY KEY (conversation_id, user_id)
);

-- ============================================================
-- MESSAGES
-- ============================================================
CREATE TABLE messages (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    sender_id       UUID NOT NULL REFERENCES users(id) ON DELETE SET NULL,
    message_type    message_type NOT NULL DEFAULT 'text',
    content         TEXT,                             -- NULL for media messages
    media_id        UUID REFERENCES user_media(id) ON DELETE SET NULL,
    status          message_status NOT NULL DEFAULT 'sent',
    reply_to_id     UUID REFERENCES messages(id) ON DELETE SET NULL,
    is_deleted      BOOLEAN NOT NULL DEFAULT FALSE,
    deleted_at      TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT chk_content_or_media CHECK (content IS NOT NULL OR media_id IS NOT NULL),
    CONSTRAINT chk_content_length CHECK (char_length(content) <= 500)
);

-- ============================================================
-- POSTS (upload tab — photo/video posts with mood + privacy)
-- ============================================================
CREATE TABLE posts (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    caption     VARCHAR(2200),
    mood        mood_type,
    visibility  post_visibility NOT NULL DEFAULT 'public',
    is_deleted  BOOLEAN NOT NULL DEFAULT FALSE,
    deleted_at  TIMESTAMPTZ,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ============================================================
-- POST MEDIA (a post can have multiple media items)
-- ============================================================
CREATE TABLE post_media (
    post_id         UUID NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
    media_id        UUID NOT NULL REFERENCES user_media(id) ON DELETE CASCADE,
    display_order   SMALLINT NOT NULL DEFAULT 0,
    PRIMARY KEY (post_id, media_id)
);

-- ============================================================
-- BLOCKS
-- ============================================================
CREATE TABLE blocks (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    blocker_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    blocked_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    reason      VARCHAR(500),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_block UNIQUE (blocker_id, blocked_id),
    CONSTRAINT chk_no_self_block CHECK (blocker_id != blocked_id)
);

-- ============================================================
-- REPORTS (abuse reporting)
-- ============================================================
CREATE TABLE reports (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    reporter_id     UUID NOT NULL REFERENCES users(id) ON DELETE SET NULL,
    reported_user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    reported_post_id UUID REFERENCES posts(id) ON DELETE SET NULL,
    reason          report_reason NOT NULL,
    description     VARCHAR(1000),
    status          VARCHAR(20) NOT NULL DEFAULT 'pending', -- pending, reviewed, dismissed, actioned
    reviewed_by     UUID REFERENCES users(id) ON DELETE SET NULL, -- admin user
    reviewed_at     TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ============================================================
-- SUBSCRIPTIONS (premium payment tracking)
-- ============================================================
CREATE TABLE subscriptions (
    id                  UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    tier                subscription_tier NOT NULL,
    payment_provider    VARCHAR(50) NOT NULL, -- 'stripe', 'apple_iap', 'google_play'
    provider_sub_id     TEXT UNIQUE,          -- external subscription ID from provider
    status              VARCHAR(20) NOT NULL, -- 'active', 'cancelled', 'expired', 'past_due'
    current_period_start TIMESTAMPTZ NOT NULL,
    current_period_end  TIMESTAMPTZ NOT NULL,
    cancelled_at        TIMESTAMPTZ,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ============================================================
-- NOTIFICATIONS
-- ============================================================
CREATE TABLE notifications (

💘 6. Matching & Nearby Algorithm
6.1 Discover Feed Query Logic
-- The core discovery query
-- Returns profiles not yet swiped, within distance, matching preferences

SELECT
    u.id,
    u.display_name,
    DATE_PART('year', AGE(u.birth_date)) AS age,
    up.bio,
    up.current_mood,
    up.location_city,
    up.is_verified,
    ST_Distance(
        up.location::geography,
        ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)::geography
    ) / 1000 AS distance_km,
    um.cdn_url AS avatar_url,
    ARRAY_AGG(DISTINCT ui.interest) AS interests,
    us.activity_score
FROM users u
JOIN user_profiles up ON up.user_id = u.id
JOIN user_preferences upr ON upr.user_id = u.id
LEFT JOIN user_media um ON um.id = up.avatar_media_id
LEFT JOIN user_interests ui ON ui.user_id = u.id
LEFT JOIN user_stats us ON us.user_id = u.id
WHERE
    -- Basic eligibility
    u.account_status = 'active'
    AND u.setup_completed = TRUE
    AND u.deleted_at IS NULL
    AND u.id != :currentUserId

    -- Distance filter
    AND ST_DWithin(
        up.location::geography,
        ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)::geography,
        :maxDistanceMeters
    )

    -- Age preference filter
    AND DATE_PART('year', AGE(u.birth_date))
        BETWEEN :ageMin AND :ageMax

    -- Gender/interest preference filter
    AND (
        :interestedIn = 'everyone'
        OR up.gender::text = ANY(:interestedInArray)
    )

    -- Exclude already-swiped profiles
    AND u.id NOT IN (
        SELECT swiped_id FROM swipes WHERE swiper_id = :currentUserId
    )

    -- Exclude blocked (in both directions)
    AND u.id NOT IN (
        SELECT blocked_id FROM blocks WHERE blocker_id = :currentUserId
        UNION
        SELECT blocker_id FROM blocks WHERE blocked_id = :currentUserId
    )

    -- Public profiles OR private profiles that are already matches
    AND (
        up.visibility = 'public'
        OR EXISTS (
            SELECT 1 FROM matches m
            WHERE m.status = 'active'
            AND (
                (m.user_a_id = :currentUserId AND m.user_b_id = u.id) OR
                (m.user_b_id = :currentUserId AND m.user_a_id = u.id)
            )
        )
    )
GROUP BY u.id, up.bio, up.current_mood, up.location_city, up.is_verified,
         up.location, um.cdn_url, us.activity_score
ORDER BY
    -- Multi-factor ranking:
    (us.activity_score * 0.4)                -- 40%: active users rank higher
    + (1.0 / (distance_km + 1.0) * 100 * 0.35) -- 35%: proximity boost
    + (RANDOM() * 0.25)                      -- 25%: controlled randomness (prevents filter bubbles)
    DESC
LIMIT :limit
OFFSET :offset;

6.2 Match Detection (on every Like swipe)
// MatchingService.processSwipe()
async processSwipe(swiperId: string, swipedId: string, action: SwipeAction) {
  // 1. Record the swipe (upsert-safe)
  await this.prisma.swipe.create({
    data: { swiperId, swipedId, action }
  });

  if (action !== SwipeAction.LIKE && action !== SwipeAction.SUPER_LIKE) return;

  // 2. Check for mutual like (reverse swipe exists?)
  const reverseSwipe = await this.prisma.swipe.findUnique({
    where: { swiperId_swipedId: { swiperId: swipedId, swipedId: swiperId } },
  });

  if (!reverseSwipe || reverseSwipe.action === SwipeAction.PASS) return;

  // 3. It's a match! Create match + conversation atomically
  const [userA, userB] = [swiperId, swipedId].sort(); // enforce ordering constraint

  const match = await this.prisma.$transaction(async (tx) => {
    const match = await tx.match.create({
      data: {
        userAId: userA,
        userBId: userB,
        status: MatchStatus.ACTIVE,
      },
    });

    const conversation = await tx.conversation.create({
      data: {
        matchId: match.id,
        participants: {
          create: [
            { userId: swiperId },
            { userId: swipedId },
          ],
        },
      },
    });

    // Increment match counter for both users (denormalized stat)
    await tx.userStats.updateMany({
      where: { userId: { in: [swiperId, swipedId] } },
      data: { totalMatches: { increment: 1 } },
    });

    return { match, conversation };
  });

  // 4. Fire match event (WebSocket + push notification via BullMQ)
  this.eventEmitter.emit('match.created', {
    matchId: match.id,
    userIds: [swiperId, swipedId],
  });

  return match;
}

6.3 Activity Score Calculation
The activity_score drives ranking in the discover feed. It's recalculated hourly via BullMQ:

// Score formula (0-100 scale)
activityScore = (
  (daysSinceLastActive <= 1 ? 40 : daysSinceLastActive <= 7 ? 20 : 5)  // recency
  + (Math.min(profileCompleteness, 100) * 0.20)                         // profile completeness %
  + (Math.min(totalLikesReceived / 10, 20))                             // social proof
  + (isVerified ? 10 : 0)                                               // verification bonus
  + (subscriptionTier === 'premium' ? 10 : 0)                           // premium boost
)

8.4 Scaling Architecture (Production)
Traffic Flow:
Mobile Apps → CloudFront (static + CDN) → AWS ALB → ECS Fargate Pods

┌────────────────── AWS ECS Fargate ──────────────────┐
│  API Pods (auto-scale 2-20 instances, 1vCPU/2GB)   │
│  Worker Pods (BullMQ workers, 2-5 instances)         │
└──────────────────────────────────────────────────────┘

Database:
  RDS PostgreSQL 16 (db.r6g.xlarge, Multi-AZ)
  + 2x Read Replicas (analytics, feed queries)

Cache:
  ElastiCache Redis 7 (cluster mode, 3 shards)

WebSockets:
  Socket.io with Redis adapter → all API pods share same pub/sub bus
  Sticky sessions on ALB for WebSocket connections

Media:
  S3 → Lambda (Sharp processing) → S3 (processed) → CloudFront

Scaling triggers (Auto Scaling):
  - CPU > 70% → +1 pod
  - Memory > 80% → +1 pod
  - ALB RequestCountPerTarget > 1000/min → +1 pod

🧠 9. Future Enhancement Roadmap
Phase 2 — Enhanced Matching (3-6 months)
ML-Based Compatibility Score
  ├── Feature vector per user: interests (TF-IDF), activity patterns,
  │   response rate, conversation length, unmatch patterns
  ├── Collaborative filtering: "users similar to you also liked..."
  ├── Model: Serving via AWS SageMaker endpoint
  │   Input: user_a vector, user_b vector
  │   Output: compatibility_score (0.0-1.0)
  └── A/B test: 50% ML ranking vs. current formula

Voice + Video Calls
  ├── WebRTC signaling via Socket.io (offer/answer/ICE exchange)
  ├── TURN/STUN servers: Twilio Network Traversal Service
  └── Recording opt-in: S3 + CloudFront (CSAM scanning via AWS Rekognition)

Boost Feature (Premium)
  ├── POST /v1/boosts → sets boost_active=true for 30 mins
  ├── activity_score * 2x multiplier during boost window
  └── Tracked in boosts table with usage limits per tier

Phase 3 — Safety & Trust (6-9 months)
AI Content Moderation
  ├── Profile photos → AWS Rekognition (NSFW, face detection, age estimation)
  ├── Post images → same pipeline on Lambda trigger
  ├── Messages → Perspective API (toxicity scoring) for harassment detection
  └── Flagged content → human_review queue in admin dashboard

ID Verification
  ├── Integration: Stripe Identity or Persona.com
  ├── POST /v1/verification/session → creates external session
  ├── Webhook confirms → sets users.is_verified = true
  └── Display "ID Verified" badge on profile

GDPR / Privacy Compliance
  ├── POST /v1/users/me/data-export → async job, email download link
  ├── DELETE /v1/users/me → GDPR-compliant erasure (30-day purge)
  └── Consent tracking table for ToS versions

Phase 4 — Engagement & Monetization (9-12 months)
Push Notifications
  ├── Provider: Expo Push Notifications API (wraps FCM + APNS)
  ├── Tokens stored: push_tokens table (user_id, token, platform, active)
  ├── Dispatch: BullMQ job → Expo Push API → FCM/APNS
  └── Types: match, message, like, super_like, profile_view (premium)

Feature Flags
  ├── Provider: PostHog (self-hosted) or LaunchDarkly
  ├── Usage: @FeatureFlag('ml_matching') on service methods
  └── Gradual rollout: 1% → 10% → 50% → 100%

API Performance
  ├── Response caching: Redis cache for discover feed (2-min TTL, invalidated on swipe)
  ├── Database query analysis: pg_stat_statements + auto EXPLAIN on slow queries
  ├── Connection pooling: PgBouncer (transaction mode, 100 connections → 1000 clients)
  └── Read-replica routing: analytics/feed queries → replica, writes → primary

Subscription Tiers
  ├── Free: 50 swipes/day, no super likes, basic filters
  ├── Premium: unlimited swipes, 5 super likes/day, see who liked you, boost 1x/week
  └── Premium+: all above + priority in discover, AI match insights, unlimited boosts
